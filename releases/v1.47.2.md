# v1.47.2 — Remove run_command restrictions

**Released:** 2026-04-13
**Type:** Simplification
**Audience:** All fs-mcp users

---

## TL;DR

`run_command` is now an unrestricted shell. All command blocklists and pattern guards have been removed.

---

## What Changed

| Removed | What it was |
|---|---|
| `BLOCKED_COMMANDS` | Set blocking `rm`, `kill`, `dd`, `mkfs`, `shutdown`, `reboot`, `halt`, `poweroff`, `init`, `killall`, `pkill`, `systemctl`, `shred`, `wipefs`, `rmdir`, `rmrf`, `:()`  |
| `BLOCKED_PATTERNS` | Substring checks blocking `> /dev/`, `>/dev/`, `chmod -R 777`, `chmod 777`, `mkfs.` |
| `_validate_command_safety()` | The entire validation function and its call site |

The tool description was also simplified — no more "Blocked: ..." clause.

---

## Why

This tool is a shell. Agents using fs-mcp need full access to system commands — killing processes, cleaning up files, managing services. The blocklists were training wheels that got in the way of real work.

---

## RTK Integration

Unchanged. The file-redirect detection regex (`>{1,2}\s*/(?!dev/)`) already uses a negative lookahead for `/dev/`, so:

- `cmd 2>&1 > /dev/null` — RTK rewrite still applies (not a file redirect)
- `cmd > /tmp/data.json` — RTK rewrite still skipped (real file redirect)

Two new test cases were added to `test_redirect_regex.py` confirming this behavior.

---

## Files Changed

| File | Change |
|---|---|
| `src/fs_mcp/server.py` | Removed `BLOCKED_COMMANDS`, `BLOCKED_PATTERNS`, `_validate_command_safety()`, cleaned up tool description |
| `test_redirect_regex.py` | Added 2 test cases for `/dev/null` + `2>&1` patterns |
| `pyproject.toml` | `version = "1.47.1"` → `"1.47.2"` |
