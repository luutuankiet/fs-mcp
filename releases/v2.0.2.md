# v2.0.2 — Process group cleanup on `run_command` timeout

Patch on top of v2.0.1. Operational safety.

## What changed

`run_command` now puts every child shell into its own process group via `Setpgid`, and the timeout path SIGKILLs the entire group instead of just the foreground process. Without this, a long-running command like `make build` that spawned subprocesses would leave orphans behind on timeout — over time, the host would accumulate stale `cc1`, `go build`, `npm install` etc. processes the operator didn't ask for.

Background `&` jobs still work the same in the happy path (command exits naturally → backgrounded jobs continue under init, like any POSIX shell). The change only affects the timeout path: if the timeout fires, the *entire* tree dies, including anything the user backgrounded inside the command. Want a job to truly outlive the tool call? Use `setsid cmd &` to escape into a fresh process group.

## Verified

```bash
# Inside a probe with timeout_sec=1:
run_command "sleep 60"
# Before v2.0.2: sleep 60 orphaned, kept running for 59s after the tool call.
# After  v2.0.2: sleep 60 reaped within ~10ms of the timeout firing.
```

`pgrep -af "sleep 60"` after the probe returns nothing — confirmed in repeated runs.

No tool contract changes; this is pure operational hygiene under the hood.
